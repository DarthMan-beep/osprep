# Docker

## Зошто постои Docker

Пишувате програма, таа работи беспрекорно на вашиот лаптоп, му ја давате на колега (или
ја распоредувате на сервер) и таа се расипува. Верзијата на Python е различна, недостасува
системска библиотека, променлива на околината не е поставена, базата на податоци е
конфигурирана на друг начин. Ова е проблемот **„работи на мојата машина“**: вашата програма
не работи изолирано — таа зависи од цела околина околу неа, а таа околина се разликува од
машина до машина.

Одговорот на Docker е **контејнерот** (container): начин да се спакува апликацијата *заедно
со* сè што ѝ е потребно за да работи — извршното опкружување, библиотеките, системските
алатки, конфигурацијата — во една самостојна целина. Таа целина работи на ист начин на која
било машина што има Docker, бидејќи машината повеќе не ја обезбедува околината; контејнерот
ја носи својата сопствена.

Менталниот модел: наместо да испраќате *инструкции* за тоа како да се постави околината
(„инсталирај Java 17, потоа...“), вие ја испраќате самата околина, веќе поставена.

## Виртуелни машини наспроти контејнери

Очигледниот постар начин да се добие репродуцибилна околина е **виртуелна машина (VM)**:
извршување на цел гостински оперативен систем врз вашиот вистински. Тоа функционира, но е
тешко — секоја VM носи цел ОС (свое јадро, свој процес на подигнување, гигабајти диск).

**Контејнерот** е полесен. Тој **не** подига свое јадро. Наместо тоа, сите контејнери на еден
домаќин **го делат јадрото на домаќинот** и се изолирани еден од друг со помош на функции на
јадрото (namespaces за „што можете да видите“, cgroups за „колку можете да користите“). Ова
се нарекува **виртуелизација на ниво на ОС**. Контејнерот пакува само апликацијата и нејзините
зависности од корисничкиот простор — не цел оперативен систем.

| | Виртуелна машина | Контејнер |
|---|---|---|
| Виртуелизира | Хардвер (цел гостински ОС) | ОС (го дели јадрото на домаќинот) |
| Содржи | Цел ОС + апликација | Само апликација + нејзините зависности |
| Големина | Гигабајти | Мегабајти |
| Подигнување | Секунди до минути (подига ОС) | Милисекунди до секунди |
| Изолација | Силна (одделно јадро) | Полесна (споделено јадро) |
| Дополнителен трошок | Тежок | Лесен |

Компромисот: VM-ите даваат посилна изолација (одделно јадро) по висока цена; контејнерите
даваат *доволно добра* изолација за повеќето апликации, многу поевтино, така што можете да
извршувате многу повеќе од нив на ист хардвер.

> **Замка:** бидејќи контејнерите го делат јадрото на домаќинот, на Linux контејнер му треба
> Linux јадро. На Windows и macOS, Docker Desktop тивко извршува мала Linux VM во позадина и
> ги стартува вашите контејнери внатре во неа. Значи „контејнерите не користат VM“ е точно на
> Linux домаќин, но на Windows/macOS постои една споделена лесна VM одоздола.

## Слика наспроти контејнер

Ова е најважната разлика во Docker.

- **Слика** (image) е **шаблон само за читање**: замрзнат, спакуван датотечен систем плус
  метаподатоци (која команда да се изврши, кои порти, кои променливи на околината). Се гради
  еднаш и никогаш не се менува.
- **Контејнер** (container) е **извршувачка (или запрена) инстанца** на слика — слика на која
  ѝ е даден слој за запишување и процес.

Аналогијата со класа/објект е точна:

| Програмирање | Docker |
|---|---|
| Класа (нацрт) | **Слика** |
| Објект (инстанца) | **Контејнер** |
| `new Foo()` | `docker run foo` |

Една слика може да создаде **многу** контејнери, исто како што една класа може да има многу
објекти. Секој контејнер добива свој слој за запишување, така што тие не си пречат еден на
друг.

### Животен циклус на контејнер

Контејнерот поминува низ состојби:

```text
docker run   ──► created ──► running ──► (stopped/exited) ──► removed
                              ▲   │
                  docker start│   │docker stop
                              └───┘
```

- **created** — постои, но не е стартуван (ретко; `docker create`).
- **running** — неговиот главен процес е жив.
- **stopped / exited** — процесот заврши (заврши, се сруши или извршивте `docker stop`).
  Запрен контејнер сè уште постои и може да се рестартира со `docker start`.
- **removed** — избришан со `docker rm`; неговиот слој за запишување го нема засекогаш.

> **Замка:** запирањето на контејнер не е исто како негово отстранување. `docker ps` прикажува
> само *извршувачки* контејнери; запрените сè уште се таму (и сè уште користат диск) — користете
> `docker ps -a` за да ги видите. Тие се натрупуваат додека не извршите `docker rm` (или додека
> не извршите со `--rm`).

## Слоеви и кешот за градење

Сликата не е една целина — таа е **стек од слоеви**. **Секоја инструкција во Dockerfile создава
еден слој** врз претходните. Слојот ги бележи промените на датотечниот систем што таа инструкција
ги направила.

Две последици се многу важни:

**1. Слоевите се кешираат и се повторно користат.** Кога повторно градите, Docker го поминува
Dockerfile од врвот до дното. За секоја инструкција, ако ништо од она од што таа зависи не се
променило, Docker го користи **кешираниот** слој наместо повторно да ја врши работата. Првата
инструкција чии влезни податоци се промениле — и **сè по неа** — мора повторно да се изгради.

Ова води до клучно правило за подредување: **ставете ги стабилните, бавни чекори пред често
менуваните чекори.** Зависностите се менуваат ретко; кодот на вашата апликација се менува
постојано. Затоа, копирајте и инсталирајте ги зависностите *прво*, а потоа копирајте го вашиот
изворен код *последен*:

```dockerfile
# GOOD: deps cached separately from source
COPY package.json package-lock.json ./
RUN npm install          # this slow layer is reused while deps are unchanged
COPY . .                 # only this cheap layer rebuilds when you edit code
```

```dockerfile
# BAD: any code edit invalidates the install
COPY . .
RUN npm install          # re-runs on every single code change — slow
```

Во лошата верзија, уредувањето на еден ред од изворниот код го менува слојот `COPY . .`, што
го принудува `RUN npm install` да се изврши повторно секој пат.

**2. Слојот за запишување на контејнерот е минлив.** Слоевите на сликата се само за читање.
Кога извршувате контејнер, Docker додава еден тенок **слој за запишување** на врвот. Сè што
контејнерот запишува (логови, прикачени датотеки, редови во база на податоци) живее само во
тој слој — и тоа е **избришано кога контејнерот е отстранет.** За да ги задржите податоците,
мора да ги складирате надвор од контејнерот, во **волумен** (volume) (видете подолу).

## CLI на docker

Командната линија е начинот на кој градите слики и извршувате контејнери. Основните работи:

| Команда | Што прави |
|---|---|
| `docker build -t name:tag .` | Гради слика од Dockerfile во `.`, ја означува како `name:tag` |
| `docker run [opts] image` | Создава + стартува контејнер од слика |
| `docker ps` | Ги наведува **извршувачките** контејнери |
| `docker ps -a` | Ги наведува **сите** контејнери (вкл. запрените) |
| `docker images` | Ги наведува локалните слики |
| `docker logs <container>` | Прикажува stdout/stderr на контејнер (`-f` за следење) |
| `docker exec -it <c> sh` | Извршува команда (тука, школка) **внатре** во извршувачки контејнер |
| `docker stop <container>` | Грациозно запира извршувачки контејнер |
| `docker rm <container>` | Отстранува запрен контејнер |
| `docker rmi <image>` | Отстранува слика |
| `docker pull <image>` | Презема слика од регистар (registry) (на пр. Docker Hub) |
| `docker push <image>` | Качува слика во регистар |
| `docker network create <n>` | Создава кориснички дефинирана мрежа |
| `docker network ls` | Ги наведува мрежите |

Важните знамиња на `docker run`:

| Знаме | Значење |
|---|---|
| `-d` | **Откачено** (Detached) — извршува во позадина, го враќа промптот |
| `-p HOST:CONTAINER` | Објавува порта (port) (мапира порта на домаќинот кон порта на контејнерот) |
| `-v SRC:DST` | Прикачува волумен или bind mount на `DST` внатре во контејнерот |
| `-e KEY=value` | Поставува променлива на околината |
| `--name myapp` | Му дава на контејнерот стабилно име (инаку Docker измислува едно) |
| `--rm` | Автоматски го отстранува контејнерот кога ќе излезе (без остаток) |
| `-it` | Интерактивно + TTY — прикачува терминал (за школки, REPL-и) |
| `--network <n>` | Го прикачува контејнерот на именувана мрежа |

Примери на повикувања:

```console
$ docker build -t webapp:1.0 .
$ docker run -d --name web -p 8080:80 webapp:1.0
$ docker ps
$ docker logs -f web
$ docker exec -it web sh
$ docker stop web && docker rm web
$ docker run --rm -it ubuntu:24.04 bash      # throwaway interactive shell
```

> **Замка:** `docker run` секогаш прави **нов** контејнер. Ако постојано го извршувате,
> натрупувате контејнери. За повторно да влезете во постоечки, користете `docker start` +
> `docker exec`, а не уште еден `docker run`.

## Dockerfile

**Dockerfile** е текстуален рецепт за градење слика. Docker го чита од врвот до дното,
извршувајќи ја секоја инструкција и потврдувајќи слој.

### Основни инструкции

| Инструкција | Кога | Цел |
|---|---|---|
| `FROM` | градење | Основната слика од која се почнува (секој Dockerfile почнува тука) |
| `WORKDIR` | градење | Поставува работен директориум за подоцнежните инструкции |
| `COPY` | градење | Копира датотеки од контекстот на градење во сликата |
| `ADD` | градење | Како COPY, плус преземање од URL и автоматско извлекување на tar архиви |
| `RUN` | градење | Извршува команда **за време на градењето** на сликата (создава слој) |
| `ENV` | градење | Поставува стандардна променлива на околината вградена во сликата |
| `ARG` | градење | Променлива при градење (се предава со `--build-arg`, не е во финалната слика) |
| `EXPOSE` | градење | **Документира** на која порта слуша апликацијата (не ја објавува) |
| `CMD` | извршување | Стандардна команда што се извршува кога контејнерот **ќе стартува** |
| `ENTRYPOINT` | извршување | Фиксната извршна датотека што контејнерот ја извршува; `CMD` ги обезбедува нејзините аргументи |

### RUN наспроти CMD — време на градење наспроти време на извршување

Ова е класичната забуна. Тие се извршуваат во сосема различни моменти:

- **`RUN`** се извршува **за време на `docker build`**, за да *ја подготви сликата* (инсталира
  пакети, компајлира код). Неговиот ефект е вграден во слој. Можете да имате многу `RUN`-ови.
- **`CMD`** не прави **ништо при градење**. Тој само ја поставува **стандардната команда што
  се извршува кога контејнерот ќе стартува**. Практично има еден `CMD` (последниот победува).

```dockerfile
RUN apt-get update && apt-get install -y python3   # happens while BUILDING
CMD ["python3", "app.py"]                           # happens when you RUN the container
```

> **Замка:** `CMD` може да се пребрише при извршување — сè што е по името на сликата во
> `docker run` го заменува (`docker run myimg echo hi` извршува `echo hi`, а не `CMD`).

### CMD наспроти ENTRYPOINT

И двете дефинираат што се извршува при стартување, но тие се комбинираат:

- **`ENTRYPOINT`** е фиксната извршна датотека — „оваа слика *е* оваа програма.“
- **`CMD`** ги обезбедува стандардните **аргументи** за таа извршна датотека (и, ако
  `ENTRYPOINT` отсуствува, тој е целата команда).

```dockerfile
ENTRYPOINT ["ping"]
CMD ["localhost"]
```

`docker run myimg` извршува `ping localhost`; `docker run myimg example.com` извршува
`ping example.com` (аргументот при извршување го заменува `CMD`, но `ENTRYPOINT` останува).
Користете `ENTRYPOINT` кога контејнерот секогаш треба да биде една специфична алатка; користете
`CMD` самостојно кога сакате лесно пребришлива стандардна команда.

Претпочитајте ја **exec формата** (`["prog", "arg"]`, JSON низа) пред **shell формата**
(`prog arg`). Exec формата ја извршува вашата програма директно како PID 1, така што таа ги
прима сигналите како `docker stop` правилно; shell формата ја обвиткува во `/bin/sh -c`, што
може да ги проголта сигналите.

### Целосно разработен Dockerfile

Мала C програма компајлирана при градење, а потоа извршена кога контејнерот ќе стартува:

```dockerfile
# 1. base image with a C compiler
FROM gcc:14

# 2. all later paths are relative to /app
WORKDIR /app

# 3. copy source into the image (build context -> image)
COPY hello.c .

# 4. compile AT BUILD TIME -> produces /app/hello, baked into a layer
RUN gcc -O2 -o hello hello.c

# 5. document the intent (purely informational here)
EXPOSE 8080

# 6. what to run WHEN A CONTAINER STARTS
CMD ["./hello"]
```

Изградете го и извршете го:

```console
$ docker build -t hello:1.0 .
$ docker run --rm hello:1.0
Hello from inside a container!
```

`gcc` се извршува еднаш, за време на градењето (чекор 4). `./hello` се извршува секој пат кога
стартувате контејнер (чекор 6).

## Објавување на порти

Контејнерот има свој мрежен namespace. Сервер што слуша на порта 80 *внатре* во контејнерот
**не** е достапен од вашиот домаќин додека не ја **објавите** портата:

```console
$ docker run -d -p 8080:80 nginx
                  │    └── port INSIDE the container
                  └─────── port ON THE HOST
```

Сега `http://localhost:8080` на вашата машина стигнува до портата 80 во контејнерот. Мапирањето
е `-p HOST:CONTAINER` — лево е домаќинот, десно е контејнерот.

`EXPOSE 80` во Dockerfile **само документира** дека апликацијата користи порта 80. Тоа **не**
отвора ништо кон домаќинот. Сепак ви треба `-p` при извршување за навистина да стигнете до
услугата.

> **Замка:** двата броја често се мешаат. `-p 3000:80` значи „домаќин 3000 → контејнер 80“,
> така што пребарувате до `localhost:3000` иако апликацијата внатре слуша на 80. Тие не мора да
> се совпаѓаат.

## Волумени — задржување на податоците живи

Потсетете се дека слојот за запишување на контејнерот се брише при `docker rm`. За да ги
**зачувате** податоците (или да споделувате датотеки со домаќинот), прикачете складирање што
живее надвор од тој слој. Два вида:

**Именуван волумен** (named volume) — Docker го управува складирањето; вие се повикувате на
него по име. Најдобар за податоци што сакате да *задржите*, но не треба да ги прелистувате
директно (бази на податоци, состојба на апликацијата):

```console
$ docker volume create dbdata
$ docker run -d -v dbdata:/var/lib/postgresql/data postgres
```

Податоците преживуваат `docker rm` и можат повторно да се прикачат на нов контејнер.

**Bind mount** — мапира специфичен **директориум на домаќинот** во контејнерот. Најдобар за
*развивање* (уредувате датотеки на домаќинот, ги гледате промените внатре) и за внесување на
конфигурација/содржина:

```console
$ docker run -d -p 8080:80 -v ./site:/usr/share/nginx/html:ro nginx
                              │      │                        └ read-only
                              │      └ path inside container
                              └ path on host (current dir / site)
```

| | Именуван волумен | Bind mount |
|---|---|---|
| Извор | Управуван од Docker | Патека на вашиот домаќин |
| Се користи за | Трајни податоци на апликацијата (бази) | Изворен код за развој, конфигурација, статични датотеки |
| Преносив | Да | Не (врзан за патеки на домаќинот) |
| Можете да уредувате на домаќинот | Не лесно | Да, директно |

Додадете **`:ro`** за да прикачите само за читање, така што контејнерот не може да го измени
изворот — корисно кога прикачувате конфигурација или содржина што апликацијата треба само да ја
чита.

> **Замка:** прикачувањето *врз* директориум што веќе имал датотеки во сликата ги крие
> датотеките на сликата на таа патека — прикачувањето има предност. Празна папка на домаќинот
> прикачена со bind mount на `/app` прави `/app` да изгледа празен внатре во контејнерот.

## Променливи на околината

Променливите на околината (environment variables) се стандардниот начин да се конфигурира
контејнер без повторно да се гради (порти, режими, акредитиви, низи за поврзување).

- **При извршување:** `-e KEY=value` во `docker run` (највисок приоритет).
- **Како стандардна вредност во сликата:** `ENV KEY=value` во Dockerfile.

```dockerfile
ENV PORT=8080
ENV MODE=production
```

```console
$ docker run -e PORT=9090 -e MODE=debug myapp   # overrides the ENV defaults
```

Апликацијата ги чита на вообичаениот начин за нејзиниот јазик — на пр. `System.getenv("PORT")`
во Java, `os.environ["PORT"]` во Python, `process.env.PORT` во Node. Контејнерот не знае ниту
го интересира како биле поставени; тој само гледа променливи на околината.

> **Замка:** вредностите на `ENV` се **вградени во слоевите на сликата** — никогаш не ставајте
> тајни (API клучеви, лозинки) во Dockerfile. Предавајте ги при извршување со `-e`, со
> `--env-file` или со механизам за тајни, така што нема да завршат испратени внатре во сликата.

## Docker Compose

Вистинските апликации ретко се еден единствен контејнер — на веб-апликацијата ѝ треба база на
податоци, можеби кеш, можеби работник (worker). Стартувањето на секој од нив рачно со точните
знамиња `-p`, `-v`, `-e`, `--network`, во точниот редослед, е заморно и склоно кон грешки.

**Docker Compose** го решава ова: ги опишувате *сите* ваши услуги во **една YAML датотека**
(`compose.yaml`) и го подигате или симнувате целиот стек со **една команда**. Compose исто така
автоматски создава споделена мрежа за услугите да можат да комуницираат меѓусебно.

Разработен пример — веб-услуга што комуницира со Postgres база на податоци:

```yaml
services:
  web:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8080:80"
    environment:
      - DB_HOST=db          # reach the database by its SERVICE NAME
      - DB_PASSWORD=secret
    depends_on:
      - db
    networks:
      - appnet

  db:
    image: postgres:16
    environment:
      - POSTGRES_PASSWORD=secret
    volumes:
      - dbdata:/var/lib/postgresql/data    # named volume, persists
    networks:
      - appnet

volumes:
  dbdata:

networks:
  appnet:
```

Клучните клучеви:

| Клуч | Значење |
|---|---|
| `services` | Контејнерите што ја сочинуваат апликацијата (по еден блок за секој) |
| `image` | Користи претходно изградена слика од регистар |
| `build` | Гради од Dockerfile наместо тоа (`context` = директориум, `dockerfile` = датотека) |
| `ports` | Објавува порти, ист `HOST:CONTAINER` како `-p` |
| `volumes` | Прикачува именувани волумени или bind mount-ови, исто како `-v` |
| `environment` | Поставува променливи на околината, исто како `-e` |
| `depends_on` | Редослед на стартување — стартувај `db` пред `web` |
| `networks` | На кои мрежи се приклучува услугата |

### DNS по име на услуга

Најкорисната функција на Compose: услугите на иста мрежа се достигнуваат меѓусебно **по име на
услуга**. Во примерот, контејнерот `web` се поврзува со базата на податоци на домаќинот **`db`**
— името на услугата — а не на IP адреса. Compose извршува мал DNS сервер што го решава секое име
на услуга во адресата на тој контејнер. Затоа променливата на околината `DB_HOST=db` погоре
едноставно функционира.

Подигнете го и симнете го стекот:

```console
$ docker compose up -d --build    # build images, start everything in the background
$ docker compose ps               # see the running services
$ docker compose logs -f web      # tail one service's logs
$ docker compose down             # stop and remove containers + network
```

`--build` принудува повторно градење на услугите што имаат `build:` секција; `-d` извршува
откачено. `down` симнува сè (додадете `-v` за да ги избришете и именуваните волумени).

> **Замка:** `depends_on` контролира само **редослед на стартување**, не подготвеност. `web`
> може да стартува пред Postgres навистина да *прифаќа врски*. За вистинска подготвеност користете
> healthcheck (`depends_on: { db: { condition: service_healthy } }`) или направете апликацијата
> да ја повторува нејзината врска.

## Вежбање

Поврзете ги концептите погоре со docker вежбите на апликацијата:

- **Вашиот прв Dockerfile** — напишете минимален `FROM` + `CMD` Dockerfile, изградете го со
  `build -t` и извршете го со `run`; набљудувајте слика што создава контејнер.
- **Сервирајте датотека преку HTTP** — извршете контејнер со веб-сервер и објавете ја неговата
  порта со `-p HOST:CONTAINER` за да можете да стигнете до неа од домаќинот.
- **Конфигурирајте контејнер со ENV** — поставете стандардна вредност со `ENV` и пребришете ја
  при извршување со `-e`, и прочитајте ја од извршувачката апликација.
- **Docker Compose: сервирајте папка со nginx** — напишете `compose.yaml` што прави bind mount
  на локална папка во nginx услуга и ја објавува нејзината порта, потоа `docker compose up -d`.
