# Конкурентност и синхронизација во Java

Нормална програма прави едно нешто во даден момент: процесорот ги извршува твоите наредби
по ред. **Конкурентна** програма има неколку независни текови на извршување — **нишки**
(threads) — од кои секоја извршува своја низа наредби, но ги делат истите податоци. Тоа
делење е целата поента (нишките можат да соработуваат евтино) и истовремено целиот проблем
(можат меѓусебно да си ги расипат податоците). Речиси сѐ во овој водич се врти околу едно
прашање: **како да им дозволиме на нишките да делат податоци без да ги расипат?**

Големата промена во начинот на размислување во однос на едно-нишковата Java е дека повеќе
не можеш да претпоставуваш дека твоите наредби се извршуваат без прекин. Речиси во која било
точка, распоредувачот (scheduler) може да паузира една нишка и да пушти друга. Ако две
нишки допираат иста променлива, *испреплетувањето* (interleaving) на нивните чекори го
одредува резултатот — а ти не го контролираш тоа испреплетување. Синхронизацијата е множеството
алатки што ти дозволуваат да повратиш токму онолку контрола колку што е потребно.

## Конкурентност наспроти паралелизам

Овие два збора често се користат заменливо, но значат различни нешта.

- **Конкурентноста** (concurrency) се однесува на *структурата*: неколку задачи се *во тек*
  истовремено. На едно процесорско јадро тие се менуваат по ред — распоредувачот извршува
  парче од нишка A, ја паузира, извршува парче од нишка B, и така натаму. Ова брзо менување
  (**испреплетување**) прави тие да *изгледаат* истовремени иако во секој миг се извршува
  само една инструкција.
- **Паралелизмот** (parallelism) се однесува на *извршувањето*: неколку задачи буквално се
  извршуваат *во истиот миг*, на различни процесорски јадра.

| | Конкурентност (1 јадро) | Паралелизам (N јадра) |
|---|---|---|
| Што се случува | Нишките се испреплетуваат, се менуваат по ред | Нишките се извршуваат навистина истовремено |
| Изгледа како | Истовремено | Е истовремено |
| Бара синхронизација? | **Да** | **Да** |

Клучната поента за испит: **и двете бараат синхронизација.** Со испреплетување, нишка може
да биде паузирана среде ажурирање, а друга нишка може да набљудува полу-завршена состојба. Со
вистински паралелизам, две јадра можат да запишат во иста променлива во истиот миг. И во двата
случаи, делените податоци можат да се расипат, па затоа заштитата што ја пишуваш е иста во
двата случаи.

## Процес наспроти нишка

Програма што се извршува е **процес**. Оперативниот систем дава на секој процес своја
приватна меморија (неговиот адресен простор). Два процеси не можат случајно да читаат или
запишуваат во променливите на другиот — тие се изолирани.

**Нишка** е тек на извршување *внатре* во процес. Еден процес може да има многу нишки, и тие
ја **делат истата меморија**: ист heap, исти објекти, исти статички полиња. Секоја нишка има
само свој *call stack* (своите локални променливи и синџирот на повици на методи) и свој
програмски бројач (program counter).

| | Процес | Нишка |
|---|---|---|
| Меморија | Приватна, изолирана | **Делена** внатре во процесот |
| Создадена од ОС како | Тешка (heavyweight) | Лесна (lightweight) |
| Комуникација | Цевки (pipes), сокети, датотеки (експлицитно) | Само читање/запишување во делени објекти |
| Една што паѓа (crash) | Не ги допира другите | Може да ја расипе делената состојба за сите |

Делената меморија е токму она што ги прави нишките моќни: пренесувањето податоци меѓу нив е
евтино колку и пристап до поле. Истовремено тоа е токму она што ги прави опасни: нема ѕид што
би спречил две нишки да запишат во исто поле во истиот момент. **Делениот heap е изворот и на
моќта и на опасноста.** Синхронизацијата е начинот на кој ги поставуваме ѕидовите што
недостасуваат, селективно.

## Нишки во Java

Постојат два класични начина да се дефинира што треба да прави една нишка.

**1. Имплементирај `Runnable`** (препорачано — твојата задача е само работа, не *е-еден*
`Thread`):

```java
class Worker implements Runnable {
    public void run() {
        System.out.println("running in " + Thread.currentThread().getName());
    }
}

Thread t = new Thread(new Worker());
t.start();
```

**2. Наследи `Thread`** (полесно за пишување, но ја врзува твојата задача за класата `Thread`):

```java
class Worker extends Thread {
    public void run() {
        System.out.println("running in " + getName());
    }
}

new Worker().start();
```

`Runnable` генерално се претпочита: ја остава твојата класа слободна да наследи нешто друго, и
работи директно со пулови на нишки (thread pools) и ламбди (`new Thread(() -> doWork())`).

### `start()` наспроти `run()` — класична грешка

```java
Thread t = new Thread(task);
t.run();    // BUG: runs task in the CURRENT thread, no new thread created
t.start();  // CORRECT: asks the JVM to create a new thread, which then calls run()
```

`run()` е само обичен метод. Ако сам го повикаш, неговото тело се извршува **на нишката што
повикува**, синхроно, како кој било повик на метод. Само `start()` бара од JVM/ОС да подигне
нова нишка, која *потоа* го повикува `run()` наместо тебе. Ако твоите „нишки“ изгледа како да
се извршуваат строго една по друга, речиси сигурно си повикал `run()` наместо `start()`.

> **Замка:** смееш да повикаш `start()` на даден `Thread` објект **еднаш**. Повторното
> повикување фрла `IllegalThreadStateException`. Направи нов `Thread` за повторно да ја
> извршиш задачата.

### Чекање, спиење и daemon нишки

```java
Thread t = new Thread(task);
t.start();
t.join();            // BLOCK here until t finishes (join = "wait for it")

Thread.sleep(1000);  // pause the CURRENT thread ~1 second; does NOT release any locks
```

- **`join()`** прави *нишката што повикува* да чека додека целната нишка не заврши. На овој
  начин `main` чека своите работници (workers) пред да ги прочита нивните резултати.
- **`Thread.sleep(ms)`** ја паузира нишката што го повикува. Забележи дека е `static` метод —
  секогаш ја засега тековната нишка, никогаш некоја друга. Може да фрли
  `InterruptedException`, па мора да биде фатен или декларуран.
- **daemon нишка** е позадински помошник (`t.setDaemon(true)` *пред* `start()`). JVM
  излегува кога ќе останат само daemon нишки — таа **не** ги чека. Користи daemon нишки за
  несуштинска позадинска работа; користи нормални (кориснички) нишки за работа што мора да
  заврши.

## Стечени услови (race conditions) и атомичност

**Стечен услов** (race condition) е кога точноста на резултатот зависи од времето/редоследот
по кој се извршуваат нишките. Учебничката причина е операција што не е **атомична**.
„Атомична“ значи *неделива* — или се случува целосно или воопшто не, без шанса друга нишка да
набљудува полу-завршена состојба.

Стапица: `count++` *изгледа* атомично, но всушност се три чекори — **читај-измени-запиши**
(read-modify-write):

```text
1. READ   the current value of count from memory into a register
2. MODIFY  add 1 in the register
3. WRITE  store the new value back to memory
```

Две нишки можат да се испреплетат меѓу тие чекори и да **изгубат ажурирање**:

```java
class Counter {
    int value = 0;
    void increment() { value++; }   // NOT atomic!
}
```

```java
Counter c = new Counter();
Runnable job = () -> { for (int i = 0; i < 100_000; i++) c.increment(); };

Thread a = new Thread(job);
Thread b = new Thread(job);
a.start(); b.start();
a.join();  b.join();

System.out.println(c.value);   // EXPECTED 200000 — but often prints LESS
```

Зошто губи ажурирања — лошо испреплетување на едно `++`:

```text
Thread A: READ value (0)
Thread B: READ value (0)          <- both saw 0
Thread A: WRITE 1
Thread B: WRITE 1                 <- B overwrites A; one increment vanished
```

Конечниот број е недетерминистички и обично под 200000. Поправката значи да се направи
читај-измени-запиши **меѓусебно исклучива** (mutually exclusive) — само една нишка внатре во
даден момент.

## Критична секција и меѓусебно исклучување

**Критична секција** (critical section) е парче код што пристапува до делени податоци и не
смее да биде извршувано од повеќе од една нишка истовремено (на пр. телото на `increment()`).

**Меѓусебно исклучување** („mutex“) е гаранцијата дека **најмногу една нишка е внатре во
критичната секција во кој било момент.** Секоја нишка што сака да влезе мора прво да заземе
брава (lock); оној што ја држи бравата ја извршува секцијата додека сите други чекаат; оној
што ја држи ја ослободува бравата на излегување, и следната нишка продолжува. Таа
сериjализација е она што ги елиминира лошите испреплетувања погоре.

## Монитори и `synchronized`

Java го вградува меѓусебното исклучување директно во јазикот преку концептот **монитор**:
**секој објект има вградена (intrinsic) брава** (наречена и *монитор-брава*). Клучниот збор
`synchronized` ја зазема таа брава при влез и ја ослободува при излез — дури и ако се фрли
исклучок (exception).

**Synchronized метод** — заклучува на `this` (објектот-примач):

```java
class Counter {
    private int value = 0;
    synchronized void increment() { value++; }   // holds this's lock
    synchronized int get()        { return value; }
}
```

**Synchronized блок** — заклучува на објект што го именуваш; пофино-гранулиран:

```java
class Counter {
    private int value = 0;
    private final Object lock = new Object();

    void increment() {
        synchronized (lock) {   // only this critical section is protected
            value++;
        }
    }
}
```

**Static synchronized** — нема `this`, па затоа заклучува на **`Class` објектот** (една брава
делена од сите инстанци), штитејќи статички полиња:

```java
class Registry {
    private static int total = 0;
    static synchronized void add() { total++; }   // locks Registry.class
}
```

> **Замка — единствената најчеста грешка во синхронизацијата:** *сите* нишки што допираат
> дадено парче делени податоци мора да се синхронизираат на **истата** брава. `synchronized`
> *запишувач* не прави ништо ако *читачот* чита без синхронизација, или се синхронизира на
> различен објект. Бравата е договор; работи само ако сите се согласат да ја користат.
> Избирањето на истиот објект-брава и за читања и за запишувања е задолжително.

Бидејќи секој монитор е **повторно-влезен** (reentrant), нишка што веќе ја држи бравата на
објект може да влезе во друг `synchronized` регион на *истиот* објект без да се самозаклучи во
мртва точка.

## Брави од `java.util.concurrent`

Клучниот збор `synchronized` е практичен, но крут (бравата се ослободува точно на крајот на
блокот). Пакетот `java.util.concurrent.locks` нуди експлицитни брави со повеќе контрола.

### `ReentrantLock`

`ReentrantLock` ја врши истата работа како `synchronized`, но ти сам ги повикуваш `lock()` и
`unlock()` — па затоа **мора** да ја ослободиш во `finally` блок, инаку исклучок ќе ја
истечи (leak) бравата и засекогаш ќе ги замрзне сите други нишки.

```java
import java.util.concurrent.locks.ReentrantLock;

class Counter {
    private final ReentrantLock lock = new ReentrantLock();
    private int value = 0;

    void increment() {
        lock.lock();
        try {
            value++;          // critical section
        } finally {
            lock.unlock();    // ALWAYS release, even on exception
        }
    }
}
```

Во споредба со `synchronized`, `ReentrantLock` додава `tryLock()` (не блокирај ако не можеш да
ја добиеш), прекинливо заклучување (interruptible locking) и опции за праведност (fairness).
Цената е што мора да го запомниш `try/finally`.

### `AtomicInteger` — бројачи без брава

За едноставниот случај на бројач постои дури и подобра алатка. Класите од
`java.util.concurrent.atomic` користат хардверска инструкција **спореди-и-замени (CAS,
compare-and-swap)** за да го направат читај-измени-запиши атомично *без* никаква брава:

```java
import java.util.concurrent.atomic.AtomicInteger;

AtomicInteger value = new AtomicInteger(0);
value.incrementAndGet();   // atomic ++ , returns the new value
value.get();               // atomic read
```

Ова е **без брава** (lock-free): нема брава за заземање, нишките никогаш не се блокираат
меѓусебно, а сепак инкрементот е атомичен. Претпочитај го за бројачи од една променлива;
посегни по брава кога мора да направиш *повеќе* операции атомични заедно.

## Бројачки семафори

**Семафор** е бројач на достапни **дозволи** (permits). Тоа е вистинската алатка кога сакаш да
дозволиш влез на *до N* нишки истовремено, наместо точно една.

```java
import java.util.concurrent.Semaphore;

Semaphore sem = new Semaphore(3);   // 3 permits

void useResource() throws InterruptedException {
    sem.acquire();        // take a permit; BLOCK if none are available
    try {
        // at most 3 threads run this region at the same time
        doWork();
    } finally {
        sem.release();    // give the permit back — ALWAYS, even on exception
    }
}
```

Како работи:

- `acquire()` го намалува бројот на дозволи; ако бројот е веќе 0, нишката се блокира додека
  некој не ослободи дозвола.
- `release()` го зголемува бројот и буди нишка што чека, ако има таква.

| Дозволи | Се однесува како | Ефект |
|---|---|---|
| `new Semaphore(1)` | **бинарен семафор ≈ mutex** | меѓусебно исклучување: една наеднаш |
| `new Semaphore(N)` | **бројачки семафор** | ограничува конкурентност на N нишки |

> **Замка:** секогаш `release()` во `finally`. Истечена (leaked) дозвола трајно го намалува
> базенот — по доволно истекувања, `acquire()` блокира засекогаш и системот застанува. Исто
> така забележи дека семафорот нема поим за сопственост: која било нишка може да ослободи
> дозвола, за разлика од брава.

## Координација со `wait` / `notify` / `notifyAll`

Меѓусебното исклучување спречува нишките да се судираат, но понекогаш нишка мора да **чека на
услов** што само друга нишка може да го направи вистинит — на пр. „чекај додека баферот не е
празен“. Тоа е она што го обезбедуваат `wait()` / `notify()` / `notifyAll()`. Тие се методи на
`Object` и работат *заедно* со мониторот.

Правила што ги прават да работат (и гласно паѓаат ако се игнорираат):

- Смееш да ги повикаш само **додека ја држиш бравата (мониторот) на објектот** (внатре во
  `synchronized` на тој објект). Инаку добиваш `IllegalMonitorStateException`.
- **`wait()`** атомично *ја ослободува бравата* и ја успива нишката. Ова е клучно: им
  овозможува на *други* нишки да ја заземат бравата и да го променат условот. Кога нишката ќе
  биде разбудена, таа **повторно ја зазема бравата** пред да се врати од `wait()`.
- **`notify()`** буди *една* нишка што чека; **`notifyAll()`** ги буди *сите*. Будењето не ја
  предава бравата — разбудената нишка (или нишки) сепак мора повторно да ја заземе.

> **Замка — секогаш чекај во `while` јамка, никогаш во `if`:** нишка може да се разбуди без
> условот да е вистинит (бидејќи друга нишка ја добила трката, или поради *лажно будење*
> (spurious wakeup) дозволено од JVM). Откако `wait()` ќе се врати, мора **повторно да го
> провериш условот** и да чекаш повторно ако сѐ уште не важи. `notifyAll` е безбедниот
> стандард: со `notify` може да ја разбудиш „погрешната“ нишка и да оставиш една што може да
> работи заспана.

### Производител–потрошувач со ограничен бафер (канонскиот пример)

Производител (producer) додава елементи; потрошувач (consumer) ги отстранува. Баферот има
фиксен капацитет, па затоа: **put чека додека баферот е полн; take чека додека е празен.**

```java
import java.util.LinkedList;
import java.util.Queue;

class BoundedBuffer<T> {
    private final Queue<T> queue = new LinkedList<>();
    private final int capacity;

    BoundedBuffer(int capacity) { this.capacity = capacity; }

    public synchronized void put(T item) throws InterruptedException {
        while (queue.size() == capacity) {   // WHILE, not if
            wait();                          // full: release lock, sleep
        }
        queue.add(item);
        notifyAll();                         // wake any waiting consumers
    }

    public synchronized T take() throws InterruptedException {
        while (queue.isEmpty()) {            // WHILE, not if
            wait();                          // empty: release lock, sleep
        }
        T item = queue.remove();
        notifyAll();                         // wake any waiting producers
        return item;
    }
}
```

Проследи го: потрошувач повикува `take()` на празен бафер, ја зема бравата, гледа `isEmpty()`,
повикува `wait()` — што *ја ослободува* бравата и спие. Производител сега може да влезе во
`put()`, да додаде елемент и да повика `notifyAll()`. Потрошувачот се буди, **повторно ја
зазема бравата**, повторно го проверува `while` (сега не е празен), го отстранува елементот и
се враќа. `while` јамката е она што го прави ова точно при лажни будења и повеќе нишки што
чекаат.

## Мртва точка (deadlock)

**Мртва точка** (deadlock) е кога две или повеќе нишки се блокирани засекогаш, секоја чекајќи
ресурс што го држи другата. Никој не може да продолжи. Мртвата точка бара **сите четири
Кофманови услови** (Coffman conditions) да важат истовремено:

| Услов | Значење |
|---|---|
| **Меѓусебно исклучување** | Ресурс се држи на неделив начин (еден држач во даден момент) |
| **Држи и чекај** (hold and wait) | Нишка држи еден ресурс додека чека на друг |
| **Без преземање** (no preemption) | Ресурс не може насилно да се одземе; само држачот го ослободува |
| **Кружно чекање** (circular wait) | Циклус од нишки, секоја чека на ресурсот на следната |

Скрши **кој било еден** услов и мртвата точка станува невозможна.

### Класичната мртва точка со две брави во спротивен редослед

```java
Object lockA = new Object();
Object lockB = new Object();

// Thread 1
synchronized (lockA) {
    synchronized (lockB) { /* ... */ }   // wants B while holding A
}

// Thread 2
synchronized (lockB) {
    synchronized (lockA) { /* ... */ }   // wants A while holding B
}
```

Ако нишка 1 ја грабне `lockA` и нишка 2 ја грабне `lockB` во истиот момент, секоја потоа чека
на бравата што ја држи другата — кружно чекање — и двете блокираат засекогаш.

### Поправки

- **Конзистентен глобален редослед на брави** — стандардната поправка. Одлучи единствен
  редослед за сите брави (на пр. секогаш прво заземи ја бравата на сметката со помал id) и
  направи *секоја* нишка да ги зазема во тој редослед. Без циклус во графот на чекање,
  кружното чекање е невозможно.

  ```java
  Object first  = id1 < id2 ? lockA : lockB;   // always lock the lower id first
  Object second = id1 < id2 ? lockB : lockA;
  synchronized (first) {
      synchronized (second) { /* transfer */ }
  }
  ```

- **Една единствена брава** — ако една груба (coarse) брава штити сѐ, нема втора брава на која
  да се чека, па мртва точка не може да се формира. Поедноставно, но ја намалува
  конкурентноста.

> **Поврзани опасности.** *Изгладнување* (starvation) — нишка никогаш не добива време на
> распоредувачот или никогаш не победува во борбата за брава, па не напредува додека другите
> напредуваат. *Жива точка* (livelock) — нишките не се блокирани, но постојано реагираат
> една на друга (на пр. двете постојано се повлекуваат и повторно се обидуваат) па ништо
> вистинско не се завршува. За разлика од мртвата точка, во жива точка нишките се зафатени;
> само никогаш не завршуваат.

## Вежбање

Поврзи ги концептите погоре со вежбите од апликацијата:

- **Бројач безбеден за нишки (race)** — репродуцирај ја грешката со изгубено ажурирање, потоа
  поправи ја со `synchronized`, `ReentrantLock` или `AtomicInteger`.
- **Паралелна сума на разлики (редукција)** — подели ја работата меѓу нишки и комбинирај ги
  парцијалните резултати; вежбај `join()` и избегнување на делена променлива состојба.
- **Производител–потрошувач со ограничен бафер (wait/notify)** — имплементирај `put`/`take`
  со проверките на условот во `while` јамка прикажани погоре.
- **Ограничи конкурентност со семафор** — користи `Semaphore(N)` со `acquire()`/`release()`
  во `try/finally` за да ограничиш колку нишки извршуваат секција наеднаш.
- **Пренос на пари без мртва точка (редослед на брави)** — пренеси меѓу сметки користејќи
  конзистентен глобален редослед на брави така што два спротивни преноси никогаш не можат да
  доведат до мртва точка.
