# Bash скриптирање ("командни процедури")

Скриптата е едноставна текстуална датотека што ги содржи истите команди што би ги
напишале на промптот. Кога ја извршувате, школката (shell) ја чита датотеката линија по
линија и ја извршува секоја команда токму како да сте ја напишале интерактивно. Нема чекор
на компилација — Bash е **интерпретер**, па затоа "синтаксна грешка" се открива дури кога
извршувањето ќе стигне до таа линија.

Големата ментална промена во споредба со јазик како C или Python е следнава: **Bash е прво
лансер на команди, а втор е програмски јазик.** Речиси сè што пишувате во суштина е "изврши
ја оваа програма со овие зборови како аргументи." Контролните структури (`if`, `while`,
`case`) и променливите постојат главно за да одлучат *кои* команди да се извршат и *со кои
аргументи*. Разбирањето дека "зборовите" се основната единица — и како школката претвора
линија текст во листа од зборови — објаснува речиси секое правило за наводници и стапица
подолу.

## Shebang и извршување скрипти

Првата линија на скриптата е **shebang** (`#!` проследено со патека до интерпретер):

```bash
#!/bin/bash
```

### Како shebang всушност работи

`#!` не е функционалност на Bash — тоа е функционалност на **кернелот**. Кога извршувате
датотека, кернелот ги гледа нејзините први два бајта. Ако се `#!`, кернелот *не* се обидува
да ја изврши датотеката како бинарна; наместо тоа го чита остатокот од таа линија, го
третира како патека до интерпретер и го извршува тој интерпретер, проследувајќи ѝ ја
патеката на скриптата како аргумент. Така:

```console
$ ./script.sh           # you type this
```

всушност зад кулисите станува:

```console
/bin/bash ./script.sh
```

Затоа shebang мора да биде **самата прва линија** без празна линија или празно место пред
`#!`, и затоа патеката мора да биде апсолутна — кернелот не прави никакво пребарување на
`PATH`.

- `#!/bin/bash` — го користи вистинскиот Bash интерпретер (потребно за низи, `[[ ]]`,
  трикови со `$(( ))`, `select`, итн.).
- `#!/bin/sh` — бара само POSIX-однесување на школката. На многу системи `/bin/sh` *не* е
  Bash (може да биде `dash`), па "bashizmi" како низите ќе откажат. За овој курс,
  претпочитајте `#!/bin/bash`.
- `#!/usr/bin/env bash` — преносна форма што го наоѓа `bash` преку `PATH` наместо тврдо да
  го запише `/bin/bash`. Честа во пракса; и двете се прифатливи за испитот.

### Три начини за извршување на скрипта

```console
$ chmod +x script.sh    # set the execute bit once
$ ./script.sh           # run it as a program (kernel reads the shebang)

$ bash script.sh        # run it by naming the interpreter (shebang & exec bit ignored)

$ source script.sh      # or: . script.sh
```

Разликата е важна:

| Повикување | Нов процес? | Чита shebang? | Влијае на тековната школка? |
|---|---|---|---|
| `./script.sh` | Да (детска школка) | Да | Не |
| `bash script.sh` | Да (детска школка) | Не (го именувавте bash) | Не |
| `source` / `.` | Не (се извршува во тековната школка) | Не | **Да** — промените на променливи и `cd` опстануваат |

Мора да напишете `./script.sh`, а не само `script.sh`, бидејќи школката го пребарува `PATH`
само за имиња на команди — а `.` (тековниот директориум) обично **не** е во `PATH` од
безбедносни причини. `./` кажува "токму оваа датотека, тука."

`#` започнува коментар: сè од `#` до крајот на линијата се игнорира (освен во shebang). Нема
блок-коментари.

## Променливи

Bash променливите се **бестипни** — секоја вредност во основа е **низа од знаци** (string).
Нема `int`, нема `bool`; `count=5` ја складира двознаковната низа `5`, а аритметика се случува
само кога експлицитно ќе побарате (видете [Аритметика](#аритметика)).

### Доделување: зошто нема празни места околу `=`

```bash
name="Ana"          # correct
name = "Ana"         # WRONG
```

Ова е единствената најчеста почетничка грешка, а причината е моделот на "лансер на команди".
Школката ја дели линијата на зборови по празни места. `name="Ana"` е еден збор без празно
место, па школката ја препознава формата на *доделување* `word=value`. Но `name = "Ana"` се
**три** збора: `name`, `=`, `Ana`. Школката се обидува да изврши **програма наречена `name`**
со аргументи `=` и `Ana`, давајќи `name: command not found`. Отсуството на празни места е тоа
што ѝ кажува на школката "ова е доделување, а не команда."

### Користење на променлива: `$` поттикнува експанзија

```bash
name="Ana"
echo "$name"        # use — always quote (see below)
echo "${name}_x"    # braces delimit where the name ends -> Ana_x
echo "$name_x"      # WRONG: looks for a variable called name_x (likely empty)
```

`$name` е **експанзија на параметар**: пред да се изврши командата, школката го заменува
`$name` со вредноста на променливата. Заградите во `${name}` се потребни секогаш кога знакот
по името би можел да биде дел од име (буква, цифра или долна црта).

### Наводници — основниот ментален модел

Ова е темата што најмногу луѓе ги сопнува и предизвикува најмногу `shellcheck`
предупредувања, па затоа усвојте го *редоследот на операциите*. Откако школката ќе ја замени
вредноста на променливата, изведува уште два чекори врз **необградениот** (unquoted)
резултат:

1. **Делење на зборови** (word splitting) — резултатот се сецка на одделни зборови на секое
   празно место, табулатор или нов ред (технички, на секој знак во `$IFS`, Internal Field
   Separator, чија стандардна вредност е празно место/табулатор/нов ред).
2. **Glob експанзија** (експанзија на патеки) — секој збор што содржи `*`, `?` или `[...]` се
   заменува со листата од соодветни имиња на датотеки.

Двојните наводници ги исклучуваат **двата** чекори за вредноста внатре во нив. Тоа е целата
поента на обградувањето `"$var"`.

```bash
file="my report.txt"

rm $file            # DANGER: splits into two words -> rm "my" "report.txt"
rm "$file"          # correct: one argument, "my report.txt"
```

```bash
pattern="*.txt"
echo $pattern       # glob runs: prints every .txt file in the directory
echo "$pattern"     # prints the literal string  *.txt
```

| Стил на наводници | Експанзија на променлива `$x` | Командна супституција `$(...)` | Делење зборови / globbing |
|---|---|---|---|
| `"double"` | да | да | **не** (безбедно) |
| `'single'` | **не** (целосно литерално) | **не** | не |
| необградено | да | да | **да** (опасно) |

Единечните наводници се целосно литерални — дури и `$` и `\` значат самите себе. Користете
ги кога сакате текстот точно онаков каков што е напишан:

```bash
echo 'Cost is $5 for *everyone*'   # prints literally, no expansion, no glob
```

**Правило за испитот и за `shellcheck`: обградете секоја експанзија на променлива како
`"$var"` и секоја командна супституција како `"$(cmd)"` освен ако немате конкретна причина да
сакате делење/globbing.** Намерниот исклучок се заглавијата на циклуси како `for x in $list`
каде што *сакате* делење — но дури и таму, низа е поуредна.

### Атрибути и опсег на променливи

```bash
readonly PI=3.14159   # constant; assigning to it later is an error
unset name            # removes the variable entirely
export PATH           # mark a variable for export to child processes
greeting="hi" prog    # one-shot: set greeting only for the command prog
```

Обична променлива живее **само во тековната школка** и *не* е видлива за програмите што
скриптата ги лансира. `export` ја копира во **околината** (environment), листа од `KEY=value`
низи што кернелот ги предава на секој детски процес. Така, на пример, `PATH` или `HOME`
стигнуваат до програмите што ги извршувате. Извезувањето е еднонасочно (родител → дете): дете
никогаш не може да ги промени променливите на родителот, што е токму причината зошто `source`
постои кога *навистина* сакате промените да опстанат.

Корисна форма за стандардна вредност (експанзија на параметар) ги избегнува
изненадувањата со "unbound variable":

```bash
echo "${EDITOR:-vi}"   # use $EDITOR if set & non-empty, else the literal "vi"
name="${1:?need a name}"  # exit with an error message if $1 is empty/unset
```

## Низи (Arrays)

Низите содржат индексирана листа од низи од знаци — вистинската алатка секогаш кога имате
"листа од нешта" (имиња на датотеки, аргументи) и сакате да ги избегнете опасностите од делење
зборови при набивање на листа во една низа разделена со празни места.

```bash
a=(one two three)        # create; indices 0,1,2
echo "${a[1]}"           # element 1  -> two   (0-based!)
echo "${a[@]}"           # ALL elements, each as a separate quoted word
echo "${#a[@]}"          # number of elements -> 3
echo "${!a[@]}"          # the list of indices -> 0 1 2

a[5]="six"               # arrays are sparse; gaps are allowed
a+=(four)                # append one element

unset 'a[1]'             # remove element 1 (note the quotes for shellcheck)
```

### Зошто `"${a[@]}"` е специјално

Комбинацијата на `@` **и** двојни наводници е единствениот начин да ги добиете сите елементи
назад секој како свој правилно обграден збор — зачувувајќи ги празните места внатре во
елементите:

```bash
files=("my report.txt" "notes.md")

for f in "${files[@]}"; do   # 2 iterations: "my report.txt", then "notes.md"
  echo "[$f]"
done

for f in "${files[*]}"; do   # 1 iteration: the elements joined into one string
  echo "[$f]"
done
```

`"${a[@]}"` → секој елемент е одделен збор. `"${a[*]}"` → сите елементи споени во еден збор
(разделени со првиот знак на `IFS`). Речиси секогаш сакате `@`. Необграденото `${a[@]}` би го
поделило повторно секој елемент на празни места, поништувајќи ја предноста на низата.

## Аргументи и специјални променливи

Кога извршувате `./script.sh alpha beta`, зборовите по името на скриптата стануваат
**позициски параметри**. Школката *не* парсира знаменца (flags) за вас — `$1` е едноставно
кој било збор што дошол прв.

| Пром. | Значење |
|---|---|
| `$0` | име на скриптата (онака како што е повикана) |
| `$1 … $9`, `${10}` | позициски аргументи (заградите се потребни од 10 нагоре) |
| `$#` | број на позициски аргументи |
| `$*` | сите аргументи како еден збор кога е обградено (`"$*"`) |
| `$@` | сите аргументи, секој како свој збор кога е обградено (`"$@"`) |
| `$?` | излезен статус на **последната** команда (0 = успех, не-нула = неуспех) |
| `$$` | PID на тековната школка — корисно за уникатни имиња на привремени датотеки |
| `$!` | PID на **последната позадинска** команда (`cmd &`) |
| `$-` | тековните знаменца за опции на школката |
| `$REPLY` | вредноста прочитана од `select` (и обичен `read`) |

### `"$@"` наспроти `"$*"` — разликата што е важна

```bash
# called as:  ./s.sh "a b" c
for x in "$@"; do echo "[$x]"; done   # [a b]   [c]      (2 args, preserved)
for x in "$*"; do echo "[$x]"; done   # [a b c]          (1 joined string)
```

`"$@"` се проширува точно во оригиналната листа од аргументи, со зачувани празни места внатре
во аргументите — ова е тоа што го користите за да ги **проследите сите аргументи** на друга
команда. `"$*"` споива сè во една низа. Необграденото `$@` и `$*` и двете делат зборови и
речиси секогаш се погрешни.

### Поместување на аргументи (shift)

`shift` го отфрла `$1` и ги пренумерира останатите надолу (`$2` станува `$1`, итн.), намалувајќи
го `$#`. Тоа е класичниот начин да се обработуваат аргументи во циклус:

```bash
while [ "$#" -gt 0 ]; do
  echo "arg: $1"
  shift
done
```

### Валидација на аргументи

Секогаш проверувајте `$#` пред да го допрете `$1`, инаку празен `$1` тивко ќе ја скрши
логиката подоцна:

```bash
if [ "$#" -ne 1 ]; then
  echo "usage: $0 <name>" >&2     # error messages go to stderr
  exit 1                          # non-zero exit = "I failed"
fi
echo "Hello, $1!"
```

`>&2` го пренасочува ова `echo` на стандардна грешка за да не го загадува нормалниот излез на
скриптата (што повикувачот можеби го фаќа). `exit 1` ја завршува скриптата со неуспешен статус
што *повикувачот* може да го открие преку `$?`.

## Командна супституција

Командната супституција извршува команда и го **заменува `$(...)` со стандардниот излез на
таа команда** (завршните нови редови се отстранети). Тоа е начинот на кој ги фаќате
резултатите од програмите во променливи или во средината на друга команда.

```bash
files=$(ls)                 # capture output into a variable
count=$(ls | wc -l)         # capture the result of a pipeline
today=$(date +%F)
echo "Backup-$today.tar"    # substitute inside a string
```

### Како работи зад кулисите

`$(...)` лансира **подшколка** (subshell) — одделен детски процес што ја извршува внатрешната
команда. Нејзиниот стандарден излез се фаќа во бафер наместо да оди на терминалот, а тој бафер
(минус завршните нови редови) се вметнува во командната линија *пред* да се изврши надворешната
команда. Бидејќи е подшколка, променливите што ги поставувате внатре во `$(...)` **не**
опстануваат потоа.

Постарата форма со обратни наводници `` `...` `` го прави истото, но лошо се вгнездува и потешко
се чита; `$(...)` е претпочитано и тоа е она што `shellcheck` го препорачува.

**Стапица — обградете ја и супституцијата.** Фатениот излез сè уште подлежи на делење зборови и
globbing ако е необграден:

```bash
echo "$(ls)"        # output preserved line by line
echo $(ls)          # newlines collapsed to spaces, globs expanded
```

За да фатите повеќелиниски излез во променлива, претпочитајте процесна супституција или читање
во низа; за единечна вредност, `"$(cmd)"` е безбедната форма.

## Аритметика

Бидејќи сите променливи се низи од знаци, мора да влезете во **аритметички контекст** за да
правите математика. Формата `$(( ... ))` оценува целоброен израз и го заменува нумеричкиот
резултат:

```bash
sum=$(( 2 + 3 * 4 ))     # -> 14  (normal operator precedence)
n=5
n=$(( n + 1 ))           # increment; note: no $ needed on names inside (( ))
half=$(( n / 2 ))        # INTEGER division -> 3, the remainder is discarded
rem=$(( n % 2 ))         # modulo -> 1
```

Клучни факти:

- **Само цели броеви.** Bash нема со подвижна запирка. `$(( 7 / 2 ))` е `3`. За децимали мора да
  повикате `bc` или `awk`: `result=$(echo "7/2" | bc -l)`.
- Внатре во `(( ))` може да го изоставите `$` на имињата на променливи — аритметичкиот парсер
  знае дека се променливи. `$(( n + 1 ))` и `$(( $n + 1 ))` се еквивалентни.
- `(( ... ))` **без** водечкото `$` е *команда* чиј излезен статус одразува дали резултатот бил
  не-нула — корисно во услови, и поддржува оператори во стил на C:

```bash
(( count++ ))                 # post-increment
if (( x > 10 )); then ... fi  # clean numeric comparison, no -gt needed
while (( i < n )); do ((i++)); done
```

Забележете дека `(( ))` ја следи конвенцијата за вистинитост на C: резултат `0` е "невистина"
(излезен статус 1), не-нула е "вистина" (излезен статус 0) — обратно од излезните кодови на
школката, па затоа користете го само како услов, а не за да фатите вредност.

Наследените `let` и `expr` прават слични работи, но се незгодни; претпочитајте `(( ))` / `$(( ))`.

## Излезен статус — моторот зад условите

Пред `if`, разберете го **излезниот статус**. Секоја команда, при завршување, враќа цел број
0–255 на школката: **0 значи успех**, сè друго значи неуспех. Ова е спротивно од вистинитоста
во C, и тоа е вредноста складирана во `$?`.

```bash
ls /etc >/dev/null; echo "$?"   # 0
ls /nope 2>/dev/null; echo "$?" # non-zero (e.g. 2)
```

`if`, `while`, `&&` и `||` **не** тестираат "вистина/невистина" — тие тестираат *излезен
статус*. `if cmd` го извршува `cmd` и се разгранува според тоа дали излегол со 0. Нема
специјален булов тип; "условот" е едноставно команда.

### `&&`, `||`, `!` — оператори со кратко спојување

```bash
mkdir build && cd build          # cd only if mkdir succeeded (exit 0)
ping -c1 host || echo "no reply" # echo only if ping failed (non-zero)
! grep -q error log.txt          # negate: true when grep finds nothing
```

- `A && B` — изврши `B` **само ако `A` успеал** (статусот на A бил 0).
- `A || B` — изврши `B` **само ако `A` не успеал** (статусот на A бил не-нула).
- `! A` — го инвертира излезниот статус на A.

Тие се споуваат накратко од лево кон десно. Чест идиом за "направи или умри":

```bash
[ -r "$f" ] || { echo "cannot read $f" >&2; exit 1; }
```

`{ ...; }` ги групира командите во тековната школка; забележете ги задолжителните празни места
по `{` и точката-запирка пред `}`.

## test / `[ ]`

`[` **не е синтакса** — буквално е команда (надворешна програма `/usr/bin/[`, исто така и
вградена во школката) чие друго име е `test`. `[ "$x" -gt 10 ]` е школката што ја извршува
командата `[` со аргументите `"$x"`, `-gt`, `10` и завршен `]`. `]` е едноставно задолжителен
завршен аргумент што `[` го бара. Тој единствен факт го објаснува секое правило за празни
места:

- **Празните места околу `[` и `]` се задолжителни** бидејќи ги разделуваат аргументите. `[$x]`
  би било школката да бара програма со име `[$x]`. `[ "$x" ]` е `[` со аргумент `"$x"`.
- **Празните места околу операторите се задолжителни** од истата причина: `-gt`, `=`, `-eq` се
  одделни *зборови* на аргументи, не залепени симболи. `[ "$x"=5 ]` го предава единечниот збор
  `x=5` на `[`, што не е она што го мислите.

`[` ги оценува своите аргументи како услов и **излегува со 0 ако е вистина, не-нула ако е
невистина** — па се вклопува директно во `if`/`&&`/`||`.

### Табели со оператори

| Тестови на низи | Тестови на датотеки | Тестови на цели броеви |
|---|---|---|
| `-z s` — низата е празна | `-e f` — постои | `-eq` — еднакво |
| `-n s` — низата не е празна | `-f f` — обична датотека | `-ne` — не е еднакво |
| `s1 = s2` — еднакво (низа) | `-d f` — директориум | `-lt` — помало од |
| `s1 != s2` — не е еднакво | `-r f` — читливо | `-gt` — поголемо од |
| | `-w f` — запишливо | `-le` — ≤ |
| | `-x f` — извршливо | `-ge` — ≥ |

```bash
[ -f "$f" ] && echo "exists and is a regular file"
[ -z "$name" ] && echo "name is empty"
[ "$a" = "$b" ]    && echo "strings equal"      # = for STRINGS
[ "$x" -eq "$y" ]  && echo "numbers equal"      # -eq for NUMBERS
```

### Две стапици што гризат сите

**1. Споредбата на низи наспроти цели броеви се различни оператори.** `=`/`!=` споредуваат
низи; `-eq`/`-lt`/`-gt` споредуваат броеви. Нивното мешање дава погрешни одговори:

```bash
[ "01" = "1" ]    # FALSE — different strings
[ "01" -eq "1" ]  # TRUE  — same number
```

**2. Секогаш обградувајте, инаку празните вредности ја кршат синтаксата.** Ако `$x` е празно и
необградено:

```bash
x=""
[ $x = "yes" ]     # becomes  [ = "yes" ]  -> syntax error "unary operator expected"
[ "$x" = "yes" ]   # becomes  [ "" = "yes" ] -> works, evaluates false
```

### `[[ ]]` — надградбата на Bash

Bash има и `[[ ... ]]`, што е *синтакса со клучен збор на школката* (не команда). Внатре во неа,
делењето зборови и globbing се потиснати, па затоа необградените променливи се безбедни; додава
`&&`, `||`, `<`, `>` за низи, и `=~` за регуларни изрази и `==` со glob шаблони:

```bash
if [[ "$name" == A* ]]; then echo "starts with A"; fi   # glob match
if [[ "$x" -gt 10 && "$y" -lt 5 ]]; then ...; fi         # combine conditions
if [[ "$line" =~ ^[0-9]+$ ]]; then echo "all digits"; fi  # regex
```

За испитот, обичното `[ ]` е преносната POSIX форма и тоа е она што го користат постарите OS
материјали; `[[ ]]` е побезбедната, помоќна Bash форма. Знајте ги двете. (`shellcheck` е
задоволен со која било.)

## Услови: if / elif / else

```bash
if [ "$x" -gt 10 ]; then
  echo big
elif [ "$x" -gt 0 ]; then
  echo small
else
  echo nonpositive
fi
```

Читајте го ова буквално: `if` ја извршува командата `[ "$x" -gt 10 ]`; ако таа команда излезе со
0, се извршува гранката `then`. `;` пред `then` е разделувач на команди (наместо тоа би можеле да
го ставите `then` на своја линија). `fi` го затвора блокот. Бидејќи условот е *која било команда*,
ова е сосема валидно и идиоматско:

```bash
if grep -q "fatal" log.txt; then     # no [ ] at all — grep IS the condition
  echo "found a fatal error"
fi
```

`grep -q` не печати ништо и само го поставува својот излезен статус (0 ако се совпаднал) — токму
она што `if` го сака. Завиткувањето во `[ ]` би било грешка.

## case

`case` совпаѓа вредност со **glob шаблони** (не регуларни изрази) — почисто од долг ланец
`if`/`elif` кога се разгранува според формата на низа. Школката проба секој шаблон по редослед и
го извршува првиот што се совпаѓа.

```bash
case "$1" in
  1)        echo "one" ;;
  2|3)      echo "two or three" ;;      # | means OR
  *.txt)    echo "a text file" ;;       # glob: ends in .txt
  [0-9]*)   echo "starts with a digit" ;;
  "")       echo "empty argument" ;;
  *)        echo "something else" ;;    # * is the default/catch-all
esac
```

Секоја гранка завршува со `;;`. `*)` на крајот делува како стандарден случај — ставете го
последен, бидејќи совпаѓањето запира на првиот погодок. Обградете го субјектот (`"$1"`) за
празна или вредност со празни места сè уште да работи. Шаблоните *не* се обградени — така `*.txt`
останува glob, наместо литерал.

## Циклуси: for / while / until

### `for` — итерирање низ листа од зборови

```bash
for f in "$dir"/*; do          # iterate over matching files
  [ -f "$f" ] || continue      # skip anything that is not a regular file
  echo "$f"
done
```

`for VAR in LIST` чекори низ зборовите на `LIST` еден по еден. Клучниот детал: `"$dir"/*` е
**glob** што школката го проширува во вистинските имиња на датотеки *пред* циклусот да започне.
Ако директориумот е празен, glob не се совпаѓа со ништо и (стандардно) останува литералот `*` —
оттаму чуварот `[ -f "$f" ]`. Обградете го `"$dir"` (променливата), но оставете го `*` необградено
за да може да се прошири.

`for` во стил на C е достапен во Bash за броење:

```bash
for (( i = 0; i < 5; i++ )); do echo "$i"; done   # 0 1 2 3 4
```

Чест, но малку расипнички идиом користи `seq`:

```bash
for i in $(seq 0 20); do echo "$i"; done          # 0..20 (here unquoted on purpose)
```

За чисто броење, експанзија со загради не бара надворешна програма: `for i in {0..20}`.

### `while` — повторувај додека команда продолжува да успева

```bash
while read -r line; do          # loop body runs as long as read succeeds
  echo ">> $line"
done < input.txt                # redirect the file into the loop's stdin
```

`while CMD` го повторува телото сè додека `CMD` излегува со 0. Читањето датотека линија по линија
е канонската употреба: `read` враќа успех додека има влез и не успева (не-нула) на EOF, што го
завршува циклусот. `< input.txt` по `done` ѝ ја дава датотеката на стандардниот влез на целиот
циклус, од каде што `read` влече.

**Стапици со read:** секогаш користете `read -r` (без `-r`, обратните коси црти се толкуваат и
изобличуваат — `shellcheck` предупредува за ова). Водечкото/завршното празно место се отсекува
освен ако не го испразните `IFS`: `while IFS= read -r line`.

### `until` — инверзија на while

`until CMD` циклира *додека* `CMD` не успее (т.е. се повторува додека `CMD` не успева):

```bash
until ping -c1 -W1 host >/dev/null 2>&1; do
  echo "waiting for host..."
  sleep 2
done
echo "host is up"
```

### break / continue

`break` веднаш излегува од циклусот; `continue` го прескокнува остатокот од тековната итерација и
скока на следната. И двете земаат опционален број за да прекинат/продолжат *N* нивоа на вгнездени
циклуси (`break 2`).

## select (менија)

`select` гради нумерирано текстуално мени од листа од зборови, го печати промптот во `$PS3`, го
чита изборот на корисникот, ја поставува избраната *вредност* во променливата на циклусот и
сировиот број во `$REPLY`, потоа се повторува засекогаш додека не направите `break`.

```bash
PS3="Choose: "
select f in *.txt "Quit"; do
  case "$f" in
    Quit) break ;;
    "")   echo "invalid choice, try again" ;;   # $f is empty on a bad number
    *)    cat "$f" ;;
  esac
done
```

Белешки за тоа како се однесува:

- Листата (`*.txt "Quit"`) се проширува точно како листа за `for` — glob ги произведува
  ставките на менито.
- Ако корисникот внесе број без соодветна ставка, променливата на циклусот (`f`) станува
  **празна низа** — заштитете се со `""`) како погоре.
- `select` **не** завршува самиот по себе; мора да направите `break` за да излезете (тука, на
  `Quit`).
- Чита од стандарден влез секоја итерација; `$REPLY` го содржи точно она што е напишано.

## read — добивање влез

```bash
read -r name                    # read one line of stdin into $name
read -rp "Your age: " age       # -p prints a prompt first (no newline)
read -r first rest              # split the line: first word -> $first, the rest -> $rest
read -rt 5 answer || echo "timed out"   # -t adds a timeout in seconds
```

`read` го дели својот влез во именуваните променливи на `$IFS` (празни места/табулатори). Ако има
повеќе зборови од променливи, **последната** променлива впива сè што останува. Без име на
променлива, линијата паѓа во `$REPLY`. Користете `-r` секогаш.

## Функции

Функциите групираат команди под едно име. Се однесуваат како мини-скрипти: добиваат свои
позициски параметри (`$1`, `$2`, ...), и комуницираат резултати на два начина — со **печатење на
stdout** (фатено со командна супституција) и со **статус на враќање** (цел број 0–255, преку
`return`).

```bash
to_lower() {
  echo "$1" | tr 'A-Z' 'a-z'
}

result=$(to_lower "HELLO")   # capture the printed output -> hello
echo "$result"
```

### Како работат аргументите и "вредностите за враќање"

```bash
greet() {
  local who="$1"             # 'local' keeps the variable inside the function
  echo "Hello, $who"          # this is the function's OUTPUT (stdout)
}

is_even() {
  (( $1 % 2 == 0 ))           # the LAST command's status becomes the return status
}

is_even 4 && echo "even"     # use it like any other condition
```

- Повикајте функција по име со аргументи разделени со празни места: `greet Ana`. Внатре, `$1` е
  `Ana`, `$#` е сопствениот број на аргументи на функцијата — позициските параметри на скриптата
  се засенчени додека функцијата работи.
- `return N` го поставува **излезниот статус** (мал цел број), не вредност. За да вратите
  *податоци*, направете `echo` и фатете со `$(...)`.
- Користете `local` за променливи што не треба да протекуваат во остатокот од скриптата — Bash
  променливите се глобални по дифолт, што `shellcheck` го означува како чест извор на грешки.
- Функција со исто име како команда ја засенчува; дефинирајте ги функциите пред да ги повикате
  (Bash чита од горе кон долу).

## Пренасочување (Redirection)

Секој процес започнува со три отворени дескриптори на датотеки: **0 = stdin** (влез), **1 =
stdout** (нормален излез), **2 = stderr** (излез за грешки). Пренасочувањето ги преповрзува овие
од/кон датотеки *пред* да се изврши командата.

| Синтакса | Ефект |
|---|---|
| `cmd > file` | stdout → датотека (**отсече/презапиши**) |
| `cmd >> file` | stdout → датотека (**додај**) |
| `cmd < file` | датотека → stdin |
| `cmd 2> file` | stderr → датотека |
| `cmd > out 2> err` | stdout и stderr во одделни датотеки |
| `cmd > file 2>&1` | stdout → датотека, потоа stderr → каде што stdout сега покажува |
| `cmd &> file` | Bash кратенка за `> file 2>&1` |
| `cmd >/dev/null 2>&1` | отфрли го **сиот** излез |
| `a | b` | stdout на a → stdin на b (**цевка**) |

```bash
> report.txt              # with no command: just truncate the file to empty
echo "line" >> report.txt # append a line
sort < data.txt           # feed data.txt as input to sort
```

### Зошто редоследот во `> file 2>&1` е важен

`2>&1` значи "натерај го дескрипторот на датотека 2 (stderr) да покажува на *истото место како*
fd 1 (stdout) **во овој момент**." Пренасочувањата се обработуваат од лево кон десно, па затоа:

```bash
cmd > file 2>&1     # 1) stdout -> file   2) stderr -> (same as stdout) file   GOOD
cmd 2>&1 > file     # 1) stderr -> (same as stdout = terminal)   2) stdout -> file
                    #    -> errors still go to the TERMINAL, only normal output to file
```

Класичниот `>/dev/null 2>&1` затоа се чита: "испрати го stdout во кантата за ѓубре, потоа испрати
го stderr на истата канта за ѓубре" — т.е. **замолчи сè**. `/dev/null` е специјална датотека што
отфрла сè запишано во неа и дава EOF кога се чита.

## Повикување надворешни програми

Ова е она што скриптите главно *прават*: повикуваат други програми и реагираат на нивниот излезен
статус и излез. Школката ја наоѓа програмата со пребарување на директориумите во `$PATH` (од лево
кон десно) за соодветно име на извршлива датотека. Програмите ги поврзувате со цевки и го фаќате
нивниот излез со `$(...)`:

```bash
matches=$(grep -c "ERROR" log.txt)        # capture a count
if grep -q "ERROR" log.txt; then ...; fi   # branch on whether grep matched
sorted=$(sort -u names.txt | head -5)      # pipeline inside a substitution
```

Корисни алатки-работни коњи што ќе ги комбинирате: `grep` (пребарување), `sort`, `uniq`, `wc`
(броење), `cut`/`awk` (извлекување колони), `sed`/`tr` (трансформирање текст), `find` (одење низ
датотечниот систем), `cat`/`head`/`tail`. Секоја комуницира преку stdin/stdout и излезен статус —
универзалниот "интерфејс" на школката.

**Стапица со цевки:** во `a | b`, излезниот статус на целата цевка е статусот на **последната**
команда (`b`) по дифолт. Ако `a` не успее но `b` успее, `$?` е 0. Поставете `set -o pipefail`
(видете подолу) за цевката да не успее ако *која било* фаза не успее.

## Одбранбено скриптирање: `set -euo pipefail`

Поставувањето на овие опции при врвот на скриптата ги претвора тивките неуспеси во гласни — силно
препорачлива навика и честа тема за испит:

```bash
#!/bin/bash
set -euo pipefail
```

| Опција | Ефект |
|---|---|
| `set -e` | веднаш излези ако која било команда излезе со не-нула (запира "продолжување по грешка") |
| `set -u` | третирај употреба на **непоставена** променлива како грешка (фаќа печатни грешки како `$flie`) |
| `set -o pipefail` | цевка не успева ако **која било** фаза не успее, не само последната |

Компромиси што треба да се знаат: со `-e`, команда за која *очекувате* дека може да не успее мора
да биде заштитена (`cmd || true`); со `-u`, упатувајте се на опционални променливи со стандардна
вредност (`"${VAR:-}"`). Овие не се магија, но фаќаат голема класа на реални грешки рано.

## Целосен пример — изврши и логирај Python датотеки

Демонстрира `$#`, `"$@"`, `$?`, `case`, `continue` и пренасочување заедно:

```bash
#!/bin/bash
set -euo pipefail

if [ "$#" -eq 0 ]; then
  echo "USAGE: $0 file1.py file2.py ..." >&2
  exit 1
fi

> report.txt                          # truncate the report to empty

for file in "$@"; do                  # "$@": each argument as its own word
  if [ ! -f "$file" ]; then
    echo "Warning: $file is not a file." >&2
    continue                          # skip to the next argument
  fi
  case "$file" in
    *.py)
      if python3 "$file" >/dev/null 2>&1; then   # run silently, branch on success
        echo "$file SUCCESS" >> report.txt
      else
        echo "$file FAIL" >> report.txt
      fi
      ;;
    *)
      echo "Warning: $file is not a .py file." >&2
      ;;
  esac
done

cat report.txt
```

Преглед на подвижните делови:

- `[ "$#" -eq 0 ]` — целоброен тест на бројот на аргументи; излези со порака за употреба на stderr.
- `> report.txt` — пренасочување без команда, употребено чисто за да се испразни датотеката.
- `for file in "$@"` — итерирај ги вистинските аргументи, со зачувани празни места.
- `if python3 "$file" ...; then` — забележете дека се разгрануваме на командата **директно**
  наместо да ја извршиме и потоа одделно да го проверуваме `$?`. Постариот идиом

  ```bash
  python3 "$file" >/dev/null 2>&1
  if [ $? -eq 0 ]; then ...
  ```

  работи, но е кревок: која било команда меѓу извршувањето и проверката го презапишува `$?`.
  Тестирањето на командата директно е појасно и `shellcheck`-чисто.
- `>> report.txt` — додај го секој резултат за циклусот да акумулира лог.

## Контролна листа за испит и `shellcheck`

Вежбите се проверуваат (lint) со `shellcheck`, па затоа усвојте ги овие навики — тие директно се
мапираат на неговите најчести предупредувања:

- **Обградете секоја експанзија:** `"$var"`, `"$@"`, `"$(cmd)"`, `"${arr[@]}"`. (SC2086)
- **`read -r`** секогаш; претпочитајте `while IFS= read -r line`. (SC2162)
- **Без `$` внатре во `(( ))`** доделувања и претпочитајте `(( ))` за нумерички тестови.
- **`[ ]` бара празни места** околу заградите и операторите; обградувајте ги операндите.
- **Низа `=` наспроти цел број `-eq`** — никогаш не ги вкрстувајте.
- **Не парсирајте `ls`** во скрипти; користете glob со `for f in *` наместо тоа. (SC2012)
- **Тестирајте ги командите директно** во `if`/`while` наместо преку `$?`.
